VFX-SD Gotek Indexed Rack — Correction Context for Cursor

Purpose

Fix the previously generated VFX-RACK-BUILD so it works correctly with FlashFloppy 3.29 on the Ensoniq VFX-SD.

The prior build is not correct for deployment as-is.

⸻

What Was Wrong In The Previous Build

1. Indexed filenames were invalid for FlashFloppy indexed mode

The previous build used filenames like:

DSKA0000_VFX_SD_OS_2.10.IMG
DSKA0001_Stock_Library_Disk.IMG
DSKA0002_ATW_Colorado_VFXsdDemos.HFE

That is wrong for indexed mode.

Required rule

Indexed files must be named exactly like:

DSKA0000.IMG
DSKA0001.IMG
DSKA0002.HFE
DSKA0003.HFE

No extra suffixes, no descriptive text in the filename.

FlashFloppy indexed mode resolves files by exact indexed naming.

⸻

2. IMG.CFG was used incorrectly

The previous build wrote an IMG.CFG containing Ensoniq geometry blocks like:

[::819200]
cyls = 80
heads = 2
...

That is not the descriptive label file we want for indexed display labeling.

For this project:
	•	FF.CFG with host = ensoniq is already enough for raw .IMG Ensoniq geometry
	•	.HFE files already contain their format information
	•	IMG.CFG should be used for slot labels / indexed metadata, not geometry examples copied from docs

If label support is implemented, it must be implemented in the format actually expected by FlashFloppy for indexed labels on this device/firmware. Do not paste example geometry config into IMG.CFG again.

If there is uncertainty about the exact label syntax supported on this Gotek/display combination, preserve a separate human-readable catalog and keep deployment-safe files minimal.

⸻

3. The build mixed two concerns into filenames

The previous build tried to make filenames both:
	•	valid indexed slot names
	•	human-readable labels

That does not work.

Correct approach:
	•	Keep filenames strictly indexed (DSKAxxxx.ext)
	•	Store human-readable names in a catalog and, if supported cleanly, in the proper label config

⸻

4. The previous build likely over-pruned duplicates

The duplicate report suggests many Blank###.HFE files matched a ROM image checksum, which is suspicious and should be re-verified before deletion or omission.

Cursor must not assume the prior duplicate decisions were correct.
