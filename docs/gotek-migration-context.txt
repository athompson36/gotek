Ensoniq VFX-SD Gotek Migration Context

Goal

Convert an existing Ensoniq VFX-SD Gotek USB setup from fragile FlashFloppy native-navigation mode to a robust FlashFloppy indexed rack with descriptive on-screen labels.

This context is intended for Cursor so it can safely rename, organize, and catalog the disk images without repeating the trial-and-error that already happened.

What Was Diagnosed

Original problem

The VFX-SD Gotek setup had previously worked. After mounting the USB on macOS, creating a ZIP archive on the USB itself, copying it off, and deleting the ZIP from the USB root, the Gotek could still browse files and folders but failed to load disks, reporting:
	•	file not found
	•	later confirmed as file does not exist

Root cause

The setup was using FlashFloppy native-navigation behavior with long filenames and IMAGE_A.CFG. In that mode, FlashFloppy can still display files after filesystem changes, but its internal navigation / file resolution can break when macOS rewrites directory entries or FAT allocation details.

Key confirmation

After switching to indexed mode and renaming at least one image to:
	•	DSKA0000.IMG

the image loaded successfully.

That confirms:
	•	the hardware is good
	•	the USB format is good when properly rebuilt
	•	the image format is good
	•	FlashFloppy indexed mode is the correct long-term solution

⸻

Important Conclusions
	1.	Do not use native navigation mode for this project going forward.
	•	Avoid relying on long filenames and folder browsing on the Gotek.
	•	Avoid dependence on IMAGE_A.CFG for active use.
	2.	Use FlashFloppy indexed mode.
	•	Files should be named like DSKA0000.IMG, DSKA0001.IMG, etc.
	3.	Use a catalog (and optional IMG.CFG) for human-readable slot mapping.
	•	Indexed filenames stay exactly DSKA0000.IMG, DSKA0001.HFE, etc.
	•	The generated IMG.CFG carries Ensoniq raw-.IMG geometry (by file size) and comment lines mapping blank-template slots; see scripts/build_vfx_rack.py.
	4.	Do not create ZIPs, temp files, or edit files directly on the Gotek USB.
	•	Always work from a local Mac folder.
	•	Rebuild the USB from clean local files.

⸻

Required FlashFloppy Configuration

FF.CFG should use indexed mode and Ensoniq-compatible behavior.

Required settings:

nav-mode = indexed
host = ensoniq
interface = shugart
pin34 = ready
display-type = auto
autoselect-file = yes

If existing FF.CFG contains additional settings, preserve them unless they conflict with the above.

Priority is to ensure the final file definitely includes the lines above.

⸻

IMG.CFG Strategy

The build script emits IMG.CFG with two purposes:

	1.	Ensoniq geometry for raw sector images — FlashFloppy `examples/Host/Ensoniq/IMG.CFG` stanzas such as `[::819200]` (800k) and `[::1638400]` (1.6M). These apply to `*.img` / `*.ima` / `*.dsk` by file size; Mirage-only blocks are omitted for this VFX-SD project.
	2.	Blank template reference — leading `##` comment lines list which indexed file (`DSKA####.HFE`) corresponds to which `Blanks/BlankNNN.HFE` source. `.HFE` blanks carry track layout in the HFE header; comments are for your reference, not a second geometry definition.

Do not assume unverified “slot = label” display syntax for FlashFloppy 3.29. The authoritative friendly map is `VFX_RACK_CATALOG.md` / `.csv`.

Rules
	•	Deploy `IMG.CFG` next to `FF.CFG` on the USB root (or under `FF/` if you use that layout — then both configs go there).
	•	Slot 000 remains the OS disk in the catalog.

Source Discovery Rules for Cursor

Cursor should scan gotek/ensoniq-vfx-sd/VFX-SD Backup/ recursively for disk images (.img / .IMG and .hfe / .HFE).

Include
	•	disk images in the backup root (e.g. VFX SD OS 2.10.img, Stock Library Disk.img)
	•	disk images inside subfolders such as:
	•	ATW
	•	Ensoniq
	•	Blanks — **only** for migration: at most **10** files named `Blank001.HFE` … `Blank010.HFE` (lowest N first). The rest of `Blanks/` is **reference-only** for creating new disks and is **not** copied into the rack (see `BLANKS_OMITTED.md` in the build output).

Exclude
	•	IMAGE_A.CFG from the final indexed USB build
	•	.DS_Store
	•	._*
	•	any __MACOSX artifacts
	•	ZIP files (including VFX-SD Backup.zip — keep as archive only, not part of the indexed USB)
	•	non-disk metadata files unless explicitly needed

Normalization
	•	Treat .img and .IMG equivalently when scanning; treat .hfe and .HFE equivalently
	•	Final rack output should use consistent extensions (uppercase .IMG and/or .HFE as appropriate per source format)

⸻

Slot Assignment Rules

Cursor should organize the rack using deterministic rules.

Required slot policy
	•	DSKA0000.IMG = VFX-SD OS disk

Preferred identification logic for OS disk

Match any source image whose name strongly suggests:
	•	VFX SD OS 2.10
	•	VFX-SD OS
	•	OS 2.10

If an already-renamed DSKA0000.IMG exists and is known-good, that may be preserved as the OS slot.

Remaining disks

Assign all remaining unique disk images sequentially beginning at:
	•	DSKA0001.IMG

Ordering recommendation

Use stable deterministic ordering, ideally:
	1.	root-level named library disks
	2.	ATW disks
	3.	Ensoniq disks
	4.	up to **10** Blanks/BlankNNN template disks (after all other content)
	•	All other `Blanks/*` paths are intentionally excluded from the USB rack

Or, alternatively, alphabetical by full relative path after reserving slot 000 for the OS disk.

Whichever ordering is chosen, Cursor should:
	•	apply it consistently
	•	generate a catalog documenting source path -> assigned slot -> display label

⸻

Deduplication Guidance

Cursor should detect likely duplicates before assigning slots.

Potential duplicate cases:
	•	same filename appearing in multiple folders
	•	same file size and identical checksum
	•	previously renamed DSKAxxxx.IMG files that duplicate original named source images

Recommended dedupe method

Use checksums to identify exact duplicates.

If duplicates exist:
	•	keep one canonical copy in the indexed rack
	•	note duplicates in the catalog
	•	do not silently discard without logging the decision

Safety Rules

Cursor should follow these rules strictly:
	1.	Do not modify the original backup folder destructively.
	•	Work from copies
	•	Preserve original backup files
	2.	Do not write temporary files into the USB root during organization.
	•	Build locally first
	3.	Do not include IMAGE_A.CFG in the final indexed USB build.
	•	It belongs to the fragile native-navigation workflow we are abandoning
	4.	Do not rely on Finder ordering or macOS metadata.
	•	Ignore .DS_Store, ._*, Spotlight files, and Trash files
	5.	Do not assume every disk image is unique.
	•	Verify with checksum when possible
	6.	Do not rename source files in place unless explicitly asked.
	•	Preferred approach is to copy into a new indexed build folder

⸻

USB Deployment Process

Once Cursor has built the rack locally, the deployment process should be:

Reformat USB

diskutil eraseDisk FAT32 GOTEK MBRFormat /dev/disk18

Copy build output to USB

From the final build folder, copy every indexed disk image to the USB root (use whichever extensions exist in the build, e.g. DSKA*.IMG and/or DSKA*.HFE):

cp DSKA*.IMG /Volumes/GOTEK/
cp DSKA*.HFE /Volumes/GOTEK/
cp FF.CFG /Volumes/GOTEK/
cp IMG.CFG /Volumes/GOTEK/
sync

(If one of the DSKA globs matches nothing, the shell may error; run only the lines that apply to your build.)

Catalog files (`VFX_RACK_CATALOG.md`, etc.) and `BLANKS_OMITTED.md` are optional on the USB; keep them in the repo or build folder for reference.

Clean metadata if needed

dot_clean /Volumes/GOTEK
find /Volumes/GOTEK -name "._*" -delete
rm -rf /Volumes/GOTEK/.DS_Store
rm -rf /Volumes/GOTEK/.Spotlight-V100
rm -rf /Volumes/GOTEK/.Trashes
sync

Eject

diskutil eject /dev/disk18
